# Common Terms

### **Instance** - *в контексте программирования переводится как "экземпляр объекта".* 
#### Example: 
> К примеру у нас есть окно блокнота - это один экземпляр приложения "блокнот" или один инстанс, запускаем еще один блокнот - получим два экземпляра. Аналогично и с любыми другими объектами.

# DevOps
---
**DevOps (DEVelopment OPeration)** - это набор практик для повышения эффективности процессов разработки **(Development)** и эксплуатации **(Opetaions)** программного обеспечения за счет их непрерывной интеграции и активного взаимодействия профильных специалистов с помощью инструментов автоматизации.

**Agile** - это гибкая методология управления проектами, которая включает в себя такие методики, как "Scrum", "FDD", "Lean", "экстремальное программирование" и т.д. Agile подразумевает создание проекта в несколько итераций, в конце каждой - виден результат.

**Культура (Culture)** - кросс-функциональное сотрудничество разнопрофильных специалистов и команд за счет единного информационного пространства проектного контента, открытых каналов коммуникаций и постоянного общения всех участников.

**Автоматизация (Automatization)** - использование инструментов непрерывной поставки с прогоном каждой правки кода через серию автоматизированных тестов, часто использующих облачную инфраструктуру, и последующую упаковку успешных сборок с дальнейшим перемещением на рабочий сервер с помощью автоматизированных развертываний и управления инфраструктурой как кодом через конфигурации саморазвертываемых сред.

**Бережливость (Lean)** - устранение действий с низкой полезностью и ускорение процессов, непрерывное совершенствование через регулярный ретроспективный анализ, раздельное тестирование различных инструментов, принятие поражений, возможности быстрого обнаружения проблем и их незамедлительного решения.

**Измерение (Measurement)** производительности, например, продолжительность работы пользователей с продуктом, частота появления в логах сообщений о критических ошибках - необходимы ясные и четкие критерии оценки работы, показатели эффективности процессов.

**Обмен (Sharing)** - совместная ответственность и разделение успехов, выпуск и обеспечение работы приложения осуществляются теми же людьми, что выполняли его сборку, т.е. разработчики и операционисты взаимодействуют на каждом этапе жизненного цикла приложения.

**CI/CD** - одна из практик DevOps, подразумевающая непрерывную интеграцию и доставку. Данный набор принципов предназначен для повышения удобства, частоты и надежности развертывания изменений программного обеспечения или продукта.

_Этапы CI/CD_
1. _Написание кода_ - разработчики пишут код своего модуля и проводят тестирование в ручном режиме. После этого результат работы соединяется в главной ветке с текущей версией проекта.
2. _Сборка_ - система контроля версий инициирует автоматическую сборку и последующее тестирование проекта.
3. _Ручное тестирование_ - после проверки CI-системой работоспособности тестовой версии, код передается для ручного исследования.
4. _Релиз_ - после ручного тестирования в сборку вносятся исправления. Следом проходит релиз версии кода для клиента.
5. _Развертывание_ - на этом этапе текущая (рабочая) версия кода размещается на Prod-серверах разработчика. Клиент может взаимодействовать с программой и изучать е> функции.
6. _Поддержка и мониторинг_ - продукт начинает использоваться конечными пользователями. При этом разработчики продолжают его поддерживать и проводят анализ пользовательского опыта.
7. _Планирование_ - исходя из пользовательского опыта разрабатывается новый функционал и готовится план доработок. После этого разработчик начинает написание кода - и цикл замыкается.

**Pipeline (конвейер)** - создание единой, непрерывной и максимально автоматизированной цепочки сборки.

# Git
---
**Система контроля версий (Version Control System)** - это система, записывающая изменения в файл или набор файлов в течение времени и позволяющая вернуться позже к определенной версии.

_Виды систем контроля версий_
1. Локальные системы контроля версий. Пример: RCS.
2. Централизованные системы контроля версий. Пример: CVS, Subversion, Perforce.
3. Распределенные системы контроля версий. Пример: Git, Mercurial, Bazaar, Darcs.

**Репозиторий** - это хранилище, где хранятся и поддерживаются какие-либо данные.

**GIT** - это распределенная система контроля версиями, созданная Л. Торвальдсом для управления разработкой ядра Linux в 2005 г..

**git config** - утилита в составе "git", которая позволяет просматривать и настраивать параметры, контролирующие все аспекты работы в "git", а также его внешний вид.

**git init** - создание в текущем каталоге нового подкаталога ".git", содержащего все необходимые файлы репозитория (структуру Git репозитория).

**git clone** - получение копии существующего репозитория из удаленного либо локального хранилища.

**git status** - определение текущего состояния файлов в репозитории.

**git add** - отслеживание (добавление под версионный контроль) новых файлов.

**git commit** - индексация (фиксирование) состояния файлов.

**.gitignore** - игнорирование файлов и каталогов, соответствующих заданным шаблонам.

**git diff** - отображение различий в изменениях между содержимым рабочего каталога с содержимым индекса.

**git rm** - удаление файла из рабочего каталога и из индекса (отслеживаемых файлов).

**git mv** - перемещеине/переименование файлов.

**git log** - перечисление коммитов, сделанных в репозитории в обратном хронологическом порядке.
 
**git reset / git restore** - позволяет отменить действия (индексацию, выполнить откат изменненного файла).

**git fetch** - получение данных из удаленных репозиториев.

**git pull** - получение данных из удаленных репозиториев и слияние в текущую ветку.

**git push** - отправка изменений в удаленный репозиторий.

**git tag** - просмотр списка имеющихся тегов.

**Легковесный тег** - указатель на определенный коммит.

**Аннотированный тег** - тег, хранящийся в базе данных Git как полноценный объект. Он имеет контрольную сумму, содержит имя автора, электронную почту и дату создания, имеет комментарий и может быть подписан и проверен при помощи GPG (GNU Privacy Guard).

**git branch** - создание новой ветки, и управление существующими ветками в репозитории. По умолчанию, в репозитории имеется одна ветка master/main.

**Долгоживущие ветки** - подход, при котором в репозитории используется несколько постоянно открытых веток, которые используются для разных этапов цикла разработки, при этом регулярно изменения из одной ветки сливаются в другую.

**Тематические ветки** - временная ветка, создаваемая и используемая для работы над конкретной функциональной возможностью или решения сопутствующих задач.

**Удаленные ссылки** - это ссылки (указатели) в удаленных репозиториях, включая ветки, теги и т.д.

**git checkout** - переключение на другую ветку.

**git merge** - слияние одной ветки в другую ветку.

**Конфликт слияния** - ситуация, возникающая при слиянии двух веток, когда одна и та же часть одного и того же файла изменяется по разному в двух объединяемых ветках. Маркеры конфликта слияния: <<<<< HEAD; =====; >>>>>>>>

**git mergetool** - графический инструмент для разрешения конфликтов.

**git rebase** - перенос всех коммитов из одной ветки и применение их в том же порядке в другой ветке.

**git stash** - берет измененное состояние рабочего каталога, то есть измененные отслеживаемые файлы и проиндексированные изменения, и сохраняет их в хранилище незавершенных изменений, которые можно в любое время применить обратно.

**git rerere** - повторно использовать сохраненное решение. Данная функциональная особенность позволяет Git запомнить то, как разрешена некоторая часть конфликта, чтобы в случае возникновения такого же конфликта, Git сможет его разрешить автоматически.

# Networks
---
_Компоненты сетевой инфраструктуры_
1. _Устройства_.
2. _Средства подключения_.
3. _Сервисы_.

_Средства сетевого подключения_
1. _Металлические провода в кабелях_ - данные кодируются в электрические импульсы.
2. _Оптоволоконный кабель_ - данные кодируются в световые импульсы.
3. _Беспроводная передача_ - данные кодируются при помощи электромагнитных волн радиочастотного диапазона.

**Схема физической топологии** - физическое расположение промежуточных устройств и кабельных линий.

**Схема логической топологии** - определение устройств, портов и схемы адресации.

**Локальная сеть (LAN)** - сетевая инфраструктура, предоставляющая доступ пользователям и оконечным устройствам на небольшой территории (например: домашняя сеть, сеть университета, предприятия).

**Глобальная сеть (WAN)** - сетевая инфраструктура, предоставляющая доступ к другим сетям на большой территории (например интернет-провайдер).

**Городская сеть (MAN)** - сетевая инфраструктура, которая охватывает территорию больше, чем локальная сеть, но меньше глобальной сети (например, город).

**Беспроводная локальная сеть (WLAN)** - сетевая инфраструктура, аналогичная локальной сети, при которой пользователи и оконечные устройства соединяются при помощи беспроводной связи.

**Сеть хранения данных (SAN)** - сетевая инфраструктура, разработанная для поддержки файловых серверов, хранения данных, их получения из хранилища и репликации.

**Интернет** - это объединение взаимосвязанных сетей в мировом масштабе.

**Интранет** - это частная сеть LAN и WAN, которая принадлежит организации и доступна только ее сотрудникам и прочим авторизованным лицам.

**Конвергентные сети** - это сети передачи данных, которые обеспечивают доступ к нескольким сервисам (передача данных, голоса и видео) в рамках одной сети.

_Сетевая орхитектура должна быть_
1. Отказоустойчивой.
2. Масштабируемой.
3. Безопасной.
4. Обеспечивать качество обслуживания (QoS).

**Кодирование** - это процесс преобразования данных в форму, приемлеммую для последующей передачи.

**Декодирование** - это обратный процесс кодированию, при котором информация преобразуется в исходную форму.

**Инкапсуляция** - это процесс, при котором некая дополнительная информация добавляется к текущему элементу данных. Например, добавление заголовка поверх существующего при движении данных по сетевой модели.

**Сетевой протокол** - это формат и набор правил для обмена сообщениями (данными) между устройствами.

**Протокол HTTP** - протокол уровня приложений, который управляет взаимодействием между веб-сервером и веб-клиентом.

**Протокол TCP** - транспортный протокол, управляющий отдельными сеансами связи.

**Протокол IP** - сетевой протокол, отвечающий за прием форматированных сегментов TCP, инкапсуляцию их в пакеты, присвоение им соответствующих адресов и их доставку к узлу назначения.

**Протокол Ethernet** - канальный протокол, который описывает две основные функции: связь по каналу передачи данных и физическое перемещение данных по средству подключения.


**Эталонная модель OSI** - сетевая модель, определяющая широкий список функций и сервисов, реализуемых на каждом уровне, а также взаимодействие каждого уровня с вышестоящим и нижестоящим.

_Уровни модели OSI_
1. Физический уровень - описывает электрические, функциональные и процедурные средства для активации, поддержки и деактивации физического соединения, обеспечивающего передачу битов из одного сетевого устройства в другое.
2. Канальный уровень - описывает способы обмена кадрами данных при обмене данными между устройствами по общей среде передачи данных. 
2.1. Подуровень MAC - регулирует доступ к разделяемой физической среде.
2.2. Подуровень LLC - обеспечивает обслуживание сетевого уровня.
3. Сетевой уровень - предоставляет функции для обмена отдельными частями данных по сети между указанными оконечными устройствами. 
4. Транспортный уровень - предназначен для обеспечения передачи данных от отправителя к получателю. Данный уровень определяет сервисы для сегментации, передачи и сборки данных для отдельных коммуникаций между оконечнми устройствами.
5. Сеансовый уровнеь - обеспечивает поддержкание сеанса связи, позволяя приложениям взаимодействовать между собой длительное время.
6. Уровень представления - обеспечивает преобразование протоколов и кодирование/декодирование данных.
7. Уровень приложений - содержит протоколы для обмена данными между процессами.

**Модель TCP/IP** - сетевая модель, описывающая процесс передачи данных.

_Уровни модели TCP/IP_
1. Уровень сетевого доступа (канальный) - описывает способ кодирования данных для передачи пакета данных на физическом уровне. 
2. Сетевой (межсетевой) уровень - передача данных из одной сети в другую, определение наилучшего пути через сеть.
3. Транспортный уровень - поддерживает связь между различными устройствами в разных сетях.
4. Прикладной уровень - отображает данные для пользователя, а также обеспечивает кодирование и управление сеансами связи.

**PDU** - единица данных протокола. На прикладном уровне PDU именуются как данные, на транспортном - сегмент, на сетевом - пакет, на канальном - кадр, на физическом - биты.

**IP-адрес** - логический адрес сетевого устройства, необходимый для доставки IP-пакета от источника к месту назначения.

**IP-адрес источника** - IP-адрес устройства-отправителя, изначального источника пакета.

**IP-адрес места назначения** - IP-адрес устройства-получателя, конечного места назначения пакета.

**MAC-адрес** - физический адрес сетевого устройства, необходимый для доставки кадра с одного сетевого интерфейса на другой в одной и той же сети.
